# Prisma Schema Conventions Audit

Audit of `packages/database/prisma` against the Prisma schema conventions rule (relation definitions, IDs, timestamps, indexes, unique constraints). Generated from the schema-conventions rule.

---

## Convention 1: Relations

**Rule:** Always define both sides of a relation with proper `@relation` attributes.

**Status:** ✅ **No violations.** All models define both sides of relations with `@relation` (including the Ticket self-relation `TicketReferences`).

---

## Convention 2: IDs

**Rule:** Use `@id @default(autoincrement())` for integer IDs or `@default(cuid())` for string IDs.

**Violations:**

| Model           | Field | Issue |
|-----------------|-------|--------|
| User            | `id`  | `String @id` with no `@default(cuid())` (or equivalent). |
| Session         | `id`  | `String @id` with no default. |
| Account         | `id`  | `String @id` with no default. |
| Verification    | `id`  | `String @id` with no default. |
| Organization    | `id`  | `String @id` with no default. |
| Member          | `id`  | `String @id` with no default. |
| Invitation      | `id`  | `String @id` with no default. |

**Compliant:** MemberPermission, Ticket, TicketAttachment, Comment, CommentAttachment use `@id @default(cuid())`.

**Recommendation:** Add `@default(cuid())` (or `@default(uuid())` if you prefer) to all string `@id` fields above so IDs are always generated by the database/schema and the convention is consistent.

## Convention 3: Timestamps

**Rule:** Include `createdAt` and `updatedAt` with `@default(now())` and `@updatedAt`.

### Better Auth (organization plugin) tables: Organization, Member, Invitation

Per [Better Auth organization plugin docs](https://github.com/better-auth/better-auth/blob/canary/docs/content/docs/plugins/organization.mdx) (via Context7):

- **Organization** – Official schema: `id`, `name`, `slug`, `logo?`, `metadata?`, **`createdAt`** (Date). No `updatedAt`.
- **Member** – Official schema: `id`, `userId`, `organizationId`, `role`, **`createdAt`** (Date). No `updatedAt`.
- **Invitation** – Official schema: `id`, `email`, `inviterId`, `organizationId`, `role`, `status`, **`createdAt`** (Date), `expiresAt`. No `updatedAt`.

Better Auth defines and expects only **`createdAt`** for these tables; **`updatedAt` is not in the spec** and is not set by Better Auth. Adding `@default(now())` to `createdAt` is safe and recommended so the DB can set it if the library doesn’t pass a value. Adding `updatedAt` would be an app-level extension (Better Auth won’t maintain it).

| Model        | Current state | Recommendation |
|--------------|---------------|----------------|
| Organization | `createdAt DateTime` without `@default(now())` | Add `@default(now())` to `createdAt`. Omit `updatedAt` to match Better Auth schema, or add it only if the app will maintain it. |
| Member       | `createdAt DateTime` without `@default(now())` | Add `@default(now())` to `createdAt`. Omit `updatedAt` to match Better Auth schema, or add it only if the app will maintain it. |
| Invitation   | `createdAt` has `@default(now())` | No change for timestamps; omit `updatedAt` to match Better Auth schema. |

### Other violations (non–Better Auth)

| Model             | Issue |
|-------------------|--------|
| MemberPermission  | Missing both `createdAt` and `updatedAt`. |
| TicketAttachment  | Missing both `createdAt` and `updatedAt`. |
| CommentAttachment | Missing both `createdAt` and `updatedAt`. |

**Compliant:** User, Session, Account, Verification, Ticket, Comment.

**Recommendation:** Add `createdAt DateTime @default(now())` and `updatedAt DateTime @updatedAt` to MemberPermission, TicketAttachment, and CommentAttachment. For Organization and Member, add only `@default(now())` to `createdAt` unless you want app-maintained `updatedAt`. Leave Invitation as-is for timestamps. All schema changes require a migration.

## Convention 4: Indexes

**Rule:** Add `@@index` for frequently queried fields.

**Potential violations / gaps:**

| Model   | Issue |
|---------|--------|
| Account | Consider `@@unique([providerId, accountId])` for OAuth uniqueness (see Convention 5); if you add it, an index is implied. |

**Compliant:** Session, Account, Verification, Member, MemberPermission, Invitation, **Ticket** (indexes on `userId` and `organizationId` added; see `get-tickets.ts` / `get-ticket-list.ts`), Comment, TicketAttachment, CommentAttachment have indexes on foreign keys and/or query-heavy fields.

---

## Convention 5: Unique Constraints

**Rule:** Use `@unique` or `@@unique` for fields that must be unique.

**Potential gap:**

| Model  | Issue |
|--------|--------|
| Account | Provider + account identifier should be unique per OAuth. Consider `@@unique([providerId, accountId])` (or your actual unique pair) to avoid duplicate linked accounts. |

**Compliant:** User (email), Session (token), Organization (slug), MemberPermission (memberId + resourceType), Ticket (title, slug), and other intended uniques are defined.

---

## Summary

| Convention     | Status   | Count |
|----------------|----------|--------|
| 1. Relations   | OK       | 0 violations |
| 2. IDs         | Violations | 7 models missing default on string `@id` |
| 3. Timestamps  | Violations | 3 non–Better Auth models missing both; Organization/Member need @default(now()) on createdAt; Better Auth tables omit updatedAt by design (see Convention 3) |
| 4. Indexes     | OK       | Ticket indexes added (userId, organizationId) |
| 5. Uniques     | Gaps     | 1 model (Account) may need composite unique |

**Suggested next steps:**

1. Add `@default(cuid())` to all string `@id` fields that lack a default (or keep as-is if IDs are set by the application and document that choice).
2. Add `@default(now())` to `createdAt` on Organization and Member (Better Auth tables; omit `updatedAt` per Better Auth schema). Add full `createdAt`/`updatedAt` to MemberPermission, TicketAttachment, CommentAttachment.
3. Add `@@unique([providerId, accountId])` (or equivalent) on Account if you want to enforce one account per provider per user at the schema level.

All changes require Prisma migrations (`prisma migrate dev` or equivalent).
